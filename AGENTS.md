# Repo Instructions

## Working Style

- Keep a lightweight working memory inside the repo for non-trivial tasks.
- Use [notes/session-memory.md](notes/session-memory.md) as the default scratchpad unless the task clearly deserves its own note.
- Update that note as you work so another model or person can continue without re-discovering everything.
- When interacting with live websites, take frequent screenshots and read the real on-screen state carefully before and after every important action.

## What To Record

- `Objective`: one or two lines about the current task.
- `Todo`: short pending steps only.
- `Notes`: brief findings, assumptions, blockers, or decisions.
- `Activity log`: very short milestone bullets such as:
  - navigated to `lovelacesquare.org`
  - opened GitHub code page
  - updated `controller/run-records.mjs`
  - added Windows UI automation

## How Detailed It Should Be

- Keep entries general and compact.
- Prefer milestone-level notes, not click-by-click transcripts.
- Do not dump large command outputs.
- Do not duplicate information already obvious from the code.
- If the task is finished, leave a short handoff summary and prune stale todo items.

## When To Update It

- At the start of a meaningful task.
- After major navigation or discovery work.
- After code changes that alter the plan.
- Before handing off or stopping in the middle of a task.
